# Механизм Mix of Harness && Hand-off

**В одну строку**: упаковать **Codex CLI** и **Claude Code CLI** как вызываемые через skill исполнители, чтобы MiMoCode при попадании в «цикл с низкой отдачей» мог приостановить текущий turn, а пользователь — одним действием передать работу другому harness. Плоскость управления остаётся в сессии MiMoCode, а плоскость исполнения находится в выбранном harness — эти плоскости разделены.

Поддерживаемые harness: **Codex CLI** && **Claude Code CLI**.

---

## 1. Зачем нужен Mix of Harness

Когда единственный harness сталкивается с задачей, для которой он изначально плохо подходит, он почти никогда не восстанавливается сам: Codex более оптимистичен и склонен слишком рано объявлять о завершении; Claude Code исследует подробнее, но при явных инструкциях может застрять в многократном переписывании похожих diff. Настоящий сигнал неудачи — не то, что «один шаг завершился ошибкой», а то, что **дальнейший расход token не приносит progress**: один файл снова и снова редактируется, одна команда bash повторяется без изменений, а соотношение исследования и изменений не улучшается.

Mix of Harness (далее MoH) решает проблему, превращая каждый harness в исполнитель, который MiMoCode может запустить через skill как дочерний процесс. **Детектор Try-Best** следит за состоянием текущего turn; обнаружив цикл с низкой отдачей, он приостанавливает turn и предлагает пользователю выбрать более подходящий harness для продолжения.

**Ключевая граница**: MoH **не переключает provider/model сессии**. После выбора «передать Codex CLI» или «передать Claude Code CLI» остаются исходные сессия MiMoCode и модель. Модель лишь получает указание загрузить соответствующий skill и делегировать исполнение выбранному harness. Контекст, панель задач, память и маршрутизация одобрений не требуют перестройки.

---

## 2. Устройство SKILL

Codex и Claude Code упакованы в отдельные **built-in skill** по путям `<data>/builtin_skills/local/skills/codex/` и `<data>/builtin_skills/local/skills/claude-code/`. Содержимое каталогов:

```
codex/
  SKILL.md                # Описание триггера + правила (предпочитать headless, без интерактива кроме --yolo)
  agents/openai.yaml
  references/
    recipes.md            # Частые паттерны codex exec
    windows.md            # Отдельные указания для нативного PowerShell / WSL2

claude-code/
  SKILL.md
  references/
    config.md
    flags.md
    interactive-tmux.md
    platforms.md
    print-mode.md
```

SKILL.md — точка входа, а поле `description` определяет, когда skill router его активирует. Основной принцип — **давать прямо в skill готовые исполняемые шаблоны команд CLI**. Например, skill Codex сразу предлагает:

```bash
codex exec \
  -C /path/to/repo \
  --sandbox workspace-write \
  --ask-for-approval never \
  "<TASK>"
```

вместо того чтобы заставлять модель самостоятельно искать комбинацию flags. Различия платформ (macOS/Linux, Windows PowerShell и WSL2) описаны в документах `references/`.

> **Почему отдельный skill, а не встроенный инструмент?** Детали работы harness остаются в skill и вводятся вместе с prompt модели; слой инструментов предоставляет только возможность запустить дочерний процесс. Обновления skill, различия платформ и изменения flags можно поддерживать заменой каталога skill без изменения кода.

## 3. Пять режимов MoH

Разным задачам нужны разные структуры оркестрации. Сейчас MoH поддерживает пять режимов, причём **Fallback — режим MiMoCode по умолчанию**: именно в нём автоматически подключаются детектор Try-Best и Hand-off.

### 3.1 Single

```
Task → Codex → Validator
```

Один harness запускается напрямую, после него — validator. Подходит, когда harness полностью заслуживает доверия, а границы задачи ясны.

### 3.2 Fallback (по умолчанию)

```
Task → MiMoCode
          │ неудача/застой
          ▼
        Codex / Claude Code
```

Сначала MiMoCode пытается выполнить работу сам. После сигнала неудачи/застоя пользователь выбирает другой harness. Распространённые правила:

- N последовательных сбоев инструментов одного класса
- Нет изменений файлов более X минут
- Слишком много сжатий контекста
- Результаты тестов последовательно не улучшаются
- Расход превышает 80% бюджета
- В финальном выводе нет требуемого artifact

Детектор Try-Best (см. §4) реализует те правила, которые можно оценивать в реальном времени внутри turn.

### 3.3 Pipeline

```
Исследование Claude Code
       ↓ HandoffPacket
Реализация Codex
       ↓ Patch
Ревью MiMoCode
       ↓ Findings
Исправление Codex
```

Этапы соединены последовательно, каждый выполняет наиболее подходящий harness, а контекст передаётся между соседними этапами структурированными packets. Подходит для задач с чёткими границами этапов: сначала разобраться, затем изменить; после реализации провести ревью.

### 3.4 Parallel Competition

```
               ┌→ Claude Code → Patch A ┐
Task → Fork ───┤                        ├→ Evaluator
               └→ Codex        → Patch B ┘
```

Несколько harness независимо выполняют одну задачу, после чего evaluator выбирает результат. Подходит при неясных границах, нескольких возможных подходах и желании сделать вероятностную ставку. **Требует больше ресурсов**, поэтому не используется по умолчанию.

### 3.5 Debate / Review

```
Codex Implementer → Claude Reviewer → Codex Repairer
```

Один harness предлагает решение, другой критикует его, затем первый исправляет. Подходит для изменений, чувствительных к корректности или безопасности и требующих перекрёстной проверки.

## 4. Механизм Hand-off (Try-Best HandOff)

Try-Best HandOff автоматизирует режим Fallback: он **в реальном времени внутри turn** отслеживает циклы с низкой отдачей и при срабатывании приостанавливает turn, сохраняет свидетельства и передаёт выбор пользователю.

### 4.1 Что считается «циклом с низкой отдачей»

Режимы неудачи агента кодинга имеют наблюдаемую форму в его траектории. Try-Best выбирает самые сильные сигналы:

- **Циклы и повторения**. Один файл многократно редактируется, последовательно появляются семантически близкие diff или одна команда bash без изменений повторяется после нескольких сбоев. Это сильнейший предвестник неудачи: попав в цикл, agent почти никогда не выбирается сам, и дальнейший расход token бесполезен. Достаточно **дедупликации скользящего окна** последних N tool call; embedding не нужны.
- **Разрыв между прогрессом и расходом**. Грубый сигнал progress (изменение числа успешных тестов, доля упомянутых в issue файлов, покрытая diff) сравнивается с token burn rate. Если потрачено 40% бюджета при нулевом прогрессе, harness почти наверняка не подходит; переключиться дешевле, чем ждать.
- **Потеря ориентации**. Читать файлы на этапе исследования нормально, но широкие grep на позднем этапе, повторное чтение одних больших файлов или открытие не связанных с issue каталогов означают, что agent не построил рабочую модель repo. Это измеряется отношением операций получения новой информации к операциям изменения за последние K шагов; в здоровой траектории оно должно монотонно снижаться.
- **Преждевременное объявление о завершении**. Harness говорит done, хотя тесты не запускались или их результат не был просмотрен. **Не доверяйте самостоятельно заявленному завершению**; Codex в этом отношении оптимистичнее Claude Code.

### 4.2 Механизм обнаружения (три причины срабатывания)

Текущая реализация в `packages/opencode/src/session/try-best-detector.ts` обнаруживает первые две категории сигналов тремя правилами:

| Reason | Описание | Порог по умолчанию |
|---|---|---|
| `edit_repeat` | Похожие изменения **одного файла**: diff преобразуется в набор 3-shingle, последние 12 событий edit сравниваются по **сходству Жаккара**; сходство > 0,8 считается одним совпадением | Срабатывание после ≥ 2 накопленных совпадений («третье похожее изменение») |
| `bash_retry` | Нормализованная команда bash **последовательно завершается ошибкой**, а вывод ошибки не меняется | 3 раза подряд |
| `action_streak` | Последовательные действия одного класса (`edit` или `verify`) не дают наблюдаемого улучшения | 4 раза подряд |

Команды и результаты нормализуются, чтобы избежать ложных срабатываний из-за временных меток, временных путей или случайных seed:

- Команда: `/tmp/...` → `<TMP>`, числа из 6 и более цифр → `<NUM>`, `--seed=xxx` → `<SEED>`
- Результат: дополнительно удаляются длительности `Ns / Nms / N seconds`; свыше 2000 символов сохраняется по половине начала и конца, а в середину вставляется `<TRUNCATED>`

Команды `verify` (bun/npm/pnpm/yarn test/typecheck/lint/build, pytest, cargo test, go test, make test, tsc и т. д.) участвуют в проверке `bash_retry` и одновременно учитываются в `action_streak`.

### 4.3 Приостановка и сохранение

После срабатывания любого reason `SessionProcessor.detectTryBest`:

1. **сбрасывает monitor**, предотвращая повторные срабатывания в одном turn;
2. **задаёт `ctx.blocked = true`**, чтобы processor для последующего вывода модели return `stop`, а prompt loop немедленно завершил текущий turn;
3. **записывает synthetic `TextPart`**: `text` содержит понятное человеку описание («Try-best loop detected; this turn was paused. …»), а `metadata.origin` — `kind: "try_best"`, текущие `providerID / modelID` и полный `incident` (reason + evidence). **Part является источником истины**: даже при отключении подписчика событий сессию можно восстановить сканированием parts;
4. **публикует событие `session.try_best.detected`**: TUI подписывается на него и сразу показывает dialog. Событие даёт низкую задержку, part служит резервным каналом;
5. **публикует событие metrics** `Metrics.TryBestDetected` для подсчёта частоты по моделям/reason.

### 4.4 Выбор пользователя (три варианта в dialog TUI)

После приостановки TUI открывает dialog «Try-best loop detected — turn paused», содержащий конкретные свидетельства (например, «Near-identical edits repeated 3 times in packages/opencode/src/foo.ts»). Доступны три варианта:

1. **Продолжить с Codex CLI** (`Hand off to Codex CLI`)
2. **Продолжить с Claude Code CLI** (`Hand off to Claude Code CLI`)
3. **Оставить текущую модель, но сменить стратегию** (`Continue with <model>`) — заставить исходную модель отказаться от текущего approach и перепланировать

Кандидаты **исключают семейство текущей модели** (`handoffTargets` в `packages/opencode/src/cli/cmd/tui/util/handoff.ts`):

- текущий provider — `openai` или имя модели содержит `gpt / codex` → только «Claude Code CLI»
- имя содержит `anthropic / claude` → только «Codex CLI»
- иначе → оба варианта

Одновременно проверяется регистрация соответствующего skill (`codex` / `claude-code`) в `sync.data.command`; незарегистрированный вариант отключается.

### 4.5 Протокол исполнения (оркестрированный handoff)

После выбора harness TUI **не меняет ни сессию, ни модель**. Вместо этого он отправляет `promptAsync` исходному sessionID, передавая `<system-reminder>` как ввод следующего turn (шаблон — `formatHarnessReminder`):

```
<system-reminder>
Try-best loop detection paused the previous turn: <detail>
The user explicitly selected and authorized the <harness> harness to take over the unfinished work.
You MUST load and follow the `<skill>` skill now and invoke <harness> as the primary executor …
Give the selected harness the complete user goal, relevant workspace state, the failed approach, and all remaining validation requirements. Do not include credentials, secrets, or unrelated private data.
Stay in this CLI and supervise <harness> until it completes or reaches a concrete blocker …
Inspect the harness result and workspace changes, ensure its validation is complete, and report the final outcome to the user. Do not stop after merely launching the harness.
</system-reminder>
```

Ключевые моменты:

- **Плоскость управления = исходная сессия**: панель задач, маршрутизация одобрений, контекст и память остаются в сессии MiMoCode; harness — лишь запущенный дочерний процесс.
- **Плоскость исполнения = выбранный harness**: исследование, реализация, исправление и проверка выполняются в этом процессе. System-reminder явно запрещает использовать harness «только как справочник» или делать return сразу после launch.
- **Исходная модель остаётся в работе**: загружает skill, упаковывает работу для harness, контролирует до завершения и сообщает итог пользователю. Она не отдаёт управление, а становится runtime-руководителем harness.

При выборе «оставить текущую модель, но сменить стратегию» reminder не отправляется; снимается только `ctx.blocked`, и модель перепланирует работу в следующем turn.

## 5. Конфигурация

### 5.1 Главный переключатель

- **Переменная окружения `MIMOCODE_ENABLE_TRY_BEST_HANDOFF`** (по умолчанию `true`)
  - `false` или `0` → отключает обнаружение циклов, приостановку turn и dialog handoff.
  - Определена в `packages/opencode/src/flag/flag.ts`.

### 5.2 Пороги (`experimental.try_best`)

Пороги можно по отдельности переопределить в `mimocode.json` / config:

```json
{
  "experimental": {
    "try_best": {
      "edit_window": 12,
      "edit_similarity": 0.8,
      "edit_matches": 2,
      "action_streak": 4
    }
  }
}
```

Значения:

| Key | По умолчанию | Описание |
|---|---|---|
| `edit_window` | 12 | Число последних событий edit для сравнения |
| `edit_similarity` | 0.8 | Порог сходства Жаккара (0–1); превышение считается совпадением |
| `edit_matches` | 2 | Накопленное число совпадений до срабатывания (то есть изменение N+1) |
| `action_streak` | 4 | Число последовательных `edit`/`verify` без прогресса |

Число последовательных сбоев `bash_retry` сейчас зафиксировано как 3 (`TRY_BEST_BASH_RETRIES`), опции config нет.

### 5.3 Состояние регистрации skill

Два варианта harness в dialog Hand-off требуют регистрации skills `codex` и `claude-code` в `sync.data.command`. Незарегистрированный вариант скрывается, чтобы нельзя было выбрать harness, который невозможно запустить.
